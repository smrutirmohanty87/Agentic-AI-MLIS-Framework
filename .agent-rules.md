# Agentic QA Framework - Agent Rules

## Overview
This document defines the behavior, responsibilities, and coding standards for all agents within the Agentic QA Playwright Framework. All agents must adhere to these guidelines to maintain consistency, quality, and maintainability.

---

## Agent Roles & Responsibilities

### 1. Testcase Agent

**Primary Responsibility:** Convert business requirements into structured, Excel-ready manual test cases.

**Deliverables:**
- Structured test case documents (Excel/CSV format)
- Test case ID, Title, Description, Pre-conditions
- Step-by-step action and expected result pairs
- Test data requirements and dependencies
- Priority and smoke test indicators

**Guidelines:**
- Follow standardized test case template
- Ensure test cases are independent and reusable
- Use clear, concise language understandable by QA and business stakeholders
- Include both positive and negative test scenarios
- Document test data requirements explicitly
- Map requirements to test case IDs for traceability
- Identify dependencies and prerequisites clearly

**Not Responsible For:**
- Writing automation code
- Test execution
- Environment setup

---

### 2. Automation Agent

**Primary Responsibility:** Implement automated tests using Page Object Model (POM) architecture with TypeScript and Playwright.

**Core Principles:**
- **Page Object Model (POM):** All UI interactions are encapsulated in page objects
- **Single Responsibility:** Each file has one clear purpose
- **No Selectors in Tests:** All selectors are exclusively stored in locator files
- **Reusable Methods:** Create utility functions for common operations
- **Type Safety:** Use TypeScript strictly with proper type definitions
- **Async/Await:** Use async/await pattern for all asynchronous operations

**Folder Structure Responsibilities:**

#### `src/locators/`
- Store all CSS selectors, XPath expressions, and locator strategies
- File naming: `{PageName}.locators.ts`
- Export constants with descriptive names
- Include WAI-ARIA and accessibility selectors where applicable
- Document complex locators with comments

Example structure:
```
src/locators/
├── login.locators.ts
├── dashboard.locators.ts
└── common.locators.ts
```

#### `src/pages/`
- Create page object classes for each page/module
- File naming: `{PageName}.page.ts`
- Each class should extend a BasePage (if created)
- All methods should be async
- Methods represent user interactions (not test steps)
- Never import or use test data directly; accept as parameters
- Include wait strategies and assertions logic
- Document complex workflows

Example method pattern:
```typescript
async fillLoginForm(username: string, password: string): Promise<void>
async clickLoginButton(): Promise<void>
async getErrorMessage(): Promise<string>
```

#### `tests/`
- Create test spec files using Playwright's test framework
- File naming: `{feature}.spec.ts`
- One feature/page per spec file
- Use page object instances for interactions
- Mark tests with appropriate tags (@smoke, @critical, @regression)
- Include descriptive test titles
- Use fixtures for setup/teardown
- Never hardcode selectors
- Organize tests by page/feature

#### `src/utils/`
- Create utility functions for repeated common operations
- Examples: wait functions, assertion helpers, test data generators
- File naming: `{utility-type}.utils.ts`
- Functions should be pure and stateless where possible
- Include error handling and logging

#### `src/test-data/`
- Store test data files (JSON, fixtures, constants)
- File naming: `{data-type}.data.ts`
- Use TypeScript interfaces for type safety
- Separate test data by feature/scenario
- Use environment-specific configurations

#### `src/config/`
- Centralized configuration management
- File naming: `{config-type}.config.ts`
- Environment variables handling
- Timeout configurations
- Browser/device configurations
- Base URL and endpoints
- Logging and reporting settings

**Coding Standards:**
- Use TypeScript strict mode
- Implement proper error handling (try-catch where needed)
- Log significant actions for debugging
- Use descriptive variable and method names
- Keep methods focused and small
- Maintain DRY (Don't Repeat Yourself) principle
- Use constants for magic strings and numbers
- Implement proper wait strategies (avoid hard waits)

**Test Implementation Pattern:**
1. Import page object
2. Initialize page with context
3. Perform actions using page methods
4. Assert results
5. Clean up if necessary

**Not Responsible For:**
- Writing test cases (that's Testcase Agent)
- Debugging test failures (that's Debug Agent)
- CI/CD pipeline configuration (handled separately)

### 3. Debug Agent

**Primary Responsibility:** Analyze failing Playwright tests, identify root causes, and suggest minimal, targeted fixes.

**Investigation Process:**
- Examine error messages and stack traces first
- Review test execution logs and screenshots
- Analyze network requests if applicable
- Check locator validity and visibility
- Verify test data and pre-conditions
- Review recent code changes

**Diagnosis Guidelines:**
- Distinguish between test code issues and application issues
- Identify timing/synchronization problems
- Check for flaky selectors or dynamic elements
- Verify element visibility and interactability
- Look for timing-related race conditions

**Fix Recommendations:**
- Suggest fixes to locators (prefer stable selectors)
- Recommend strategic wait strategies without hard waits
- Advise page object method improvements
- Identify missing error handling
- Suggest test data adjustments if applicable
- Point out environmental issues if detected

**Minimal Fixes Principle:**
- Avoid refactoring unrelated code
- Fix only the root cause, not symptoms
- Provide targeted code changes with context
- Include explanations of why the fix works
- Test recommendations before implementation

**Supported Debugging Scenarios:**
- Element not found errors
- Timeout errors
- Navigation failures
- State assertion failures
- Flaky test patterns
- Performance issues
- Environment-specific failures

**Not Responsible For:**
- Writing new tests
- Creating test cases
- Application feature development

## Framework Style & Standards

### Language & Runtime
- **Language:** TypeScript (strict mode)
- **Runtime:** Node.js (LTS version or higher)
- **Package Manager:** npm

### Coding Patterns
- **Async/Await:** All asynchronous operations use async/await
- **Arrow Functions:** Prefer arrow functions for callbacks
- **Type Definitions:** Explicit types for all function parameters and returns
- **Interfaces:** Use TypeScript interfaces for complex objects
- **Constants:** Define all magic strings, numbers, and timeouts as constants

### Naming Conventions
- **Files:** kebab-case (e.g., `login.page.ts`)
- **Classes:** PascalCase (e.g., `LoginPage`)
- **Methods:** camelCase (e.g., `fillLoginForm()`)
- **Constants:** UPPER_SNAKE_CASE (e.g., `LOGIN_BUTTON`)
- **Variables:** camelCase (e.g., `userName`)

### File Organization
- One exported class/function per file when possible
- Group related functionality in appropriately named folders
- Keep files under 300 lines for readability
- Use barrel exports (index.ts) for public APIs

### Code Quality
- **Comments:** Add comments for complex logic only; code should be self-documenting
- **Error Handling:** Explicit error messages and logging
- **DRY:** Extract duplicate code into utilities
- **SOLID Principles:** Apply where applicable
- **Clean Architecture:** Maintain separation of concerns

### Testing Best Practices
- **Test Independence:** Tests should not depend on each other
- **Isolation:** Each test should set up its own pre-conditions
- **Repeatability:** Tests should produce consistent results
- **Clarity:** Test names should describe what is being tested
- **Speed:** Optimize for fast feedback; use parallel execution where safe

### Performance Considerations
- Avoid hard waits (`page.waitForTimeout()`)
- Use element-based waits with reasonable timeouts
- Implement smart waiting strategies
- Parallelize test execution safely
- Monitor and log slow operations

### Locator Strategy Priority
1. **Accessibility Locators** (data-testid, role, label)
2. **Stable Selectors** (ID, class with low change frequency)
3. **Semantic HTML** (role-based selectors)
4. **CSS Selectors** (specific, efficient)
5. **XPath** (as last resort, documented with reason)

### Git & Version Control
- Commit messages should reference agent role and change type
- Feature branches follow naming: `feature/agent-type/description`
- Pull requests should include agent role and impact assessment
- Keep commits atomic and focused

### Documentation
- README with setup instructions
- CONTRIBUTING.md with guidelines
- Inline documentation for complex algorithms
- Update .agent-rules.md when roles or standards change

## Inter-Agent Communication

### Testcase Agent → Automation Agent
- Provide well-structured test case documents
- Include test data requirements
- Specify expected outcomes clearly
- Mark critical vs. nice-to-have scenarios

### Automation Agent → Debug Agent
- Provide test execution logs and stack traces
- Include locator information and selectors used
- Document recent changes to codebase
- Provide screenshots/videos when available

### Debug Agent → Automation Agent
- Provide minimal, targeted fix suggestions
- Explain root cause clearly
- Include implementation guidance
- Suggest preventive measures

## Quality Gates

### Before Committing Automation Code
- ✅ All tests pass locally
- ✅ No hardcoded selectors in test files
- ✅ Page objects follow POM pattern
- ✅ TypeScript compiles with no errors
- ✅ Code follows naming conventions
- ✅ Appropriate wait strategies used
- ✅ Test data properly managed
- ✅ Locators properly organized

### Before Running Test Suite
- ✅ Test environment is properly configured
- ✅ Required test data is available
- ✅ Browser dependencies are installed
- ✅ All configuration files are valid

## Escalation & Special Cases

### When Agents Encounter Ambiguity
1. Document the ambiguity clearly
2. Provide multiple interpretation options
3. Request clarification with specific questions
4. Proceed with most reasonable interpretation

### Framework Limitations
- Report blocking issues immediately
- Suggest tooling improvements
- Propose architectural changes with justification
- Consider long-term maintainability impact
